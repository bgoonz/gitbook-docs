<span key="925f39c3580441efbc3fdbe1fa4cdd3e"><span offset-key="925f39c3580441efbc3fdbe1fa4cdd3e:0">Overview</span></span>

<span key="1d7b16d489e349d38448d79b8e4fb80a"><span offset-key="1d7b16d489e349d38448d79b8e4fb80a:0">React lets you define components as classes or functions. Components defined as classes currently provide more features which are described in detail on this page. To define a React component class, you need to extend </span>`React.Component`<span offset-key="1d7b16d489e349d38448d79b8e4fb80a:2">:</span></span>

<span key="35ed56779b74409a9c8ca012d4d8831e">`class Welcome extends React.Component { render() { return <h1>Hello, {this.props.name}</h1>; } }`</span>

<span key="c1615f5d388746348bb5ae442c7344e8"><span offset-key="c1615f5d388746348bb5ae442c7344e8:0">The only method you </span>*must*<span offset-key="c1615f5d388746348bb5ae442c7344e8:2"> define in a </span>`React.Component`<span offset-key="c1615f5d388746348bb5ae442c7344e8:4"> subclass is called </span>`[render()](<https://reactjs.org/docs/react-component.html#render>)`<span offset-key="c1615f5d388746348bb5ae442c7344e8:6">. All the other methods described on this page are optional.</span></span>

<span key="c4888af1f50549ed8a42ee3a885264c1">**We strongly recommend against creating your own base component classes.**<span offset-key="c4888af1f50549ed8a42ee3a885264c1:1"> In React components, </span></span><a href="https://reactjs.org/docs/composition-vs-inheritance.html" class="css-4rbku5 css-1dbjc4n r-1loqt21 r-1471scf r-1otgn73 r-1i6wzkk r-lrvibr"><span data-key="2a80f8b4a3b447c6868828d3ef915a97" data-rnw-int-class="nearest_257-26400_259-26401-237__"><span data-key="3cb0cd1aaa55436baa6ed866dd7ef6ea"><span data-offset-key="3cb0cd1aaa55436baa6ed866dd7ef6ea:0">code reuse is primarily achieved through composition rather than inheritance</span></span></span></a><span key="17f66ef8ea594e3a9a0fc69a95a37bd3"><span offset-key="17f66ef8ea594e3a9a0fc69a95a37bd3:0">.</span></span>

> <span key="c6e8b1f59585406eb0ac127438242603"><span offset-key="c6e8b1f59585406eb0ac127438242603:0">Note:React doesn’t force you to use the ES6 class syntax. If you prefer to avoid it, you may use the create-react-class module or a similar custom abstraction instead. Take a look at Using React without ES6 to learn more.</span></span>

<span key="ca61b92955f54be291c048a8392ad3d1">`render()`</span>

Keeping </span>`render()`<span offset-key="4a7bde0582dd4c31b47f00b51c06846f:4"> pure makes components easier to think about.</span></span>

<span key="b2345f19d285465dbfce69db292f6610">`constructor()`</span>

<span key="0cf8095145584883a5eff16fdb66d9a1">`constructor(props)`</span>

<span key="c31533bec2c54809bd9eed851680e09f">**If you don’t initialize state and you don’t bind methods, you don’t need to implement a constructor for your React component.**</span>

<span key="773eca599e2d4fb9952d2b3ab00c472e"><span offset-key="773eca599e2d4fb9952d2b3ab00c472e:0">The constructor for a React component is called before it is mounted. When implementing the constructor for a </span>`React.Component`<span offset-key="773eca599e2d4fb9952d2b3ab00c472e:2"> subclass, you should call </span>`super(props)`<span offset-key="773eca599e2d4fb9952d2b3ab00c472e:4"> before any other statement. Otherwise, </span>`this.props`<span offset-key="773eca599e2d4fb9952d2b3ab00c472e:6"> will be undefined in the constructor, which can lead to bugs.</span></span>

<span key="629bec0bd2ed4df59ff7258795c0db89"><span offset-key="629bec0bd2ed4df59ff7258795c0db89:0">Typically, in React constructors are only used for two purposes:</span></span>

-   <img src="data:image/svg+xml;base64,PHN2ZyB2aWV3Ym94PSIwIDAgOCAxNiIgcHJlc2VydmVhc3BlY3RyYXRpbz0ieE1pZFlNaWQgbWVldCIgY2xhc3M9InItMW5mNGpibSIgc3R5bGU9ImhlaWdodDoxMnB4O3ZlcnRpY2FsLWFsaWduOm1pZGRsZTt3aWR0aDoxMnB4Ij48cGF0aCBkPSJNMCA4YzAtMi4yIDEuOC00IDQtNHM0IDEuOCA0IDQtMS44IDQtNCA0LTQtMS44LTQtNHoiIGZpbGw9ImN1cnJlbnRDb2xvciIgZmlsbC1ydWxlPSJldmVub2RkIj48L3BhdGg+PC9zdmc+" class="r-1nf4jbm" />

    <span key="ed8389aa978d4afe913b13f6528a2c0a"><span offset-key="ed8389aa978d4afe913b13f6528a2c0a:0">Initializing </span></span><a href="https://reactjs.org/docs/state-and-lifecycle.html" class="css-4rbku5 css-1dbjc4n r-1loqt21 r-1471scf r-1otgn73 r-1i6wzkk r-lrvibr"><span data-key="867de0d1f0d14895810bea8ee42a6aad" data-rnw-int-class="nearest_257-26400_259-26401-237__"><span data-key="81a71aadc534475fa65dae1c9063e341"><span data-offset-key="81a71aadc534475fa65dae1c9063e341:0">local state</span></span></span></a><span key="c5214ad6ea7441fc81701d5a54650cfa"><span offset-key="c5214ad6ea7441fc81701d5a54650cfa:0"> by assigning an object to </span>`this.state`<span offset-key="c5214ad6ea7441fc81701d5a54650cfa:2">.</span></span>

-   <img src="data:image/svg+xml;base64,PHN2ZyB2aWV3Ym94PSIwIDAgOCAxNiIgcHJlc2VydmVhc3BlY3RyYXRpbz0ieE1pZFlNaWQgbWVldCIgY2xhc3M9InItMW5mNGpibSIgc3R5bGU9ImhlaWdodDoxMnB4O3ZlcnRpY2FsLWFsaWduOm1pZGRsZTt3aWR0aDoxMnB4Ij48cGF0aCBkPSJNMCA4YzAtMi4yIDEuOC00IDQtNHM0IDEuOCA0IDQtMS44IDQtNCA0LTQtMS44LTQtNHoiIGZpbGw9ImN1cnJlbnRDb2xvciIgZmlsbC1ydWxlPSJldmVub2RkIj48L3BhdGg+PC9zdmc+" class="r-1nf4jbm" />

    <span key="5a8cb99fa8284bb0909f905da939d275"><span offset-key="5a8cb99fa8284bb0909f905da939d275:0">Binding </span></span><a href="https://reactjs.org/docs/handling-events.html" class="css-4rbku5 css-1dbjc4n r-1loqt21 r-1471scf r-1otgn73 r-1i6wzkk r-lrvibr"><span data-key="bb8751b061714158bfce88af105e84a5" data-rnw-int-class="nearest_257-26400_259-26401-237__"><span data-key="b711477c150d4e0c8497f3dc2ecfa574"><span data-offset-key="b711477c150d4e0c8497f3dc2ecfa574:0">event handler</span></span></span></a><span key="78152587caec471f8675e5678689d8a2"><span offset-key="78152587caec471f8675e5678689d8a2:0"> methods to an instance.</span></span>

<span key="216f0570fd6a443588e8b6dab26bafac"><span offset-key="216f0570fd6a443588e8b6dab26bafac:0">You </span>**should not call** `setState()`<span offset-key="216f0570fd6a443588e8b6dab26bafac:3"> in the </span>`constructor()`<span offset-key="216f0570fd6a443588e8b6dab26bafac:5">. Instead, if your component needs to use local state, </span>**assign the initial state to** `this.state`<span offset-key="216f0570fd6a443588e8b6dab26bafac:8"> directly in the constructor:</span></span>

<span key="62ed139d5465475490f6f22c5a4f57e7">`constructor(props) { super(props); // Don't call this.setState() here! this.state = { counter: 0 }; this.handleClick = this.handleClick.bind(this); }`</span>

<span key="9a1d7522ef41406cbe680eb5b7cd55de"><span offset-key="9a1d7522ef41406cbe680eb5b7cd55de:0">Constructor is the only place where you should assign </span>`this.state`<span offset-key="9a1d7522ef41406cbe680eb5b7cd55de:2"> directly. In all other methods, you need to use </span>`this.setState()`<span offset-key="9a1d7522ef41406cbe680eb5b7cd55de:4"> instead.</span></span>

<span key="93fa475d2b78443e926e2b202f6eb002"><span offset-key="93fa475d2b78443e926e2b202f6eb002:0">Avoid introducing any side-effects or subscriptions in the constructor. For those use cases, use </span>`componentDidMount()`<span offset-key="93fa475d2b78443e926e2b202f6eb002:2"> instead.</span></span>

> <span key="2335cd0b8a4745e3a109c5d0785a8655"><span offset-key="2335cd0b8a4745e3a109c5d0785a8655:0">NoteAvoid copying props into state! This is a common mistake:constructor(props) { super(props); // Don't do this! this.state = { color: props.color }; }The problem is that it’s both unnecessary (you can use this.props.color directly instead), and creates bugs (updates to the color prop won’t be reflected in the state).Only use this pattern if you intentionally want to ignore prop updates. In that case, it makes sense to rename the prop to be called initialColor or defaultColor. You can then force a component to “reset” its internal state by changing its key when necessary.Read our blog post on avoiding derived state to learn about what to do if you think you need some state to depend on the props.</span></span>

<span key="0b4d81bb376646b3a5ba7bf6bfb21631">`componentDidMount()`</span>

<span key="84b38ba8397843a18f370b5a544191ad">`componentDidMount()`</span>

<span key="0c4aa5cbffce4a9685ded462071afec2">`componentDidMount()`<span offset-key="0c4aa5cbffce4a9685ded462071afec2:1"> is invoked immediately after a component is mounted (inserted into the tree). Initialization that requires DOM nodes should go here. If you need to load data from a remote endpoint, this is a good place to instantiate the network request.</span></span>

<span key="beb20675275740b1873481760463db83"><span offset-key="beb20675275740b1873481760463db83:0">This method is a good place to set up any subscriptions. If you do that, don’t forget to unsubscribe in </span>`componentWillUnmount()`<span offset-key="beb20675275740b1873481760463db83:2">.</span></span>

<span key="1f32f17e6fc74f209cf0028dd68b9c95"><span offset-key="1f32f17e6fc74f209cf0028dd68b9c95:0">You </span>**may call** `setState()` **immediately**<span offset-key="1f32f17e6fc74f209cf0028dd68b9c95:4"> in </span>`componentDidMount()`<span offset-key="1f32f17e6fc74f209cf0028dd68b9c95:6">. It will trigger an extra rendering, but it will happen before the browser updates the screen. This guarantees that even though the </span>`render()`<span offset-key="1f32f17e6fc74f209cf0028dd68b9c95:8"> will be called twice in this case, the user won’t see the intermediate state. Use this pattern with caution because it often causes performance issues. In most cases, you should be able to assign the initial state in the </span>`constructor()`<span offset-key="1f32f17e6fc74f209cf0028dd68b9c95:10"> instead. It can, however, be necessary for cases like modals and tooltips when you need to measure a DOM node before rendering something that depends on its size or position.</span></span>

<span key="53b721f8fda1456490e70e36ae9ed77b">`componentDidUpdate()`</span>

<span key="20a6c67ce64d4427b1e3248780704a2e">`componentDidUpdate(prevProps, prevState, snapshot)`</span>

<span key="067072bcc8744807b55483d42f03b020">`componentDidUpdate()`<span offset-key="067072bcc8744807b55483d42f03b020:1"> is invoked immediately after updating occurs. This method is not called for the initial render.</span></span>

<span key="a9aaf8fa336748dda00171b6be16e9fb"><span offset-key="a9aaf8fa336748dda00171b6be16e9fb:0">Use this as an opportunity to operate on the DOM when the component has been updated. This is also a good place to do network requests as long as you compare the current props to previous props (e.g. a network request may not be necessary if the props have not changed).</span></span>

<span key="40c783d42aa344ef8903fdc855238089">`componentDidUpdate(prevProps) { // Typical usage (don't forget to compare props): if (this.props.userID !== prevProps.userID) { this.fetchData(this.props.userID); } }`</span>

<span key="05556de12be94926b90fbc1c218d0e63"><span offset-key="05556de12be94926b90fbc1c218d0e63:0">You </span>**may call** `setState()` **immediately**<span offset-key="05556de12be94926b90fbc1c218d0e63:4"> in </span>`componentDidUpdate()`<span offset-key="05556de12be94926b90fbc1c218d0e63:6"> but note that </span>**it must be wrapped in a condition**<span offset-key="05556de12be94926b90fbc1c218d0e63:8"> like in the example above, or you’ll cause an infinite loop. It would also cause an extra re-rendering which, while not visible to the user, can affect the component performance. If you’re trying to “mirror” some state to a prop coming from above, consider using the prop directly instead. Read more about </span></span><a href="https://reactjs.org/blog/2018/06/07/you-probably-dont-need-derived-state.html" class="css-4rbku5 css-1dbjc4n r-1loqt21 r-1471scf r-1otgn73 r-1i6wzkk r-lrvibr"><span data-key="0c4d718c3ba84d18b3a6daa448bfabf6" data-rnw-int-class="nearest_257-26400_259-26401-237__"><span data-key="1e668f9d0b714ad2b0cd23d0c7b629d4"><span data-offset-key="1e668f9d0b714ad2b0cd23d0c7b629d4:0">why copying props into state causes bugs</span></span></span></a><span key="a30094e2aeb949e6b668184a9c343029"><span offset-key="a30094e2aeb949e6b668184a9c343029:0">.</span></span>

<span key="37182f5b86c9421e8efc3fbde97bf431"><span offset-key="37182f5b86c9421e8efc3fbde97bf431:0">If your component implements the </span>`getSnapshotBeforeUpdate()`<span offset-key="37182f5b86c9421e8efc3fbde97bf431:2"> lifecycle (which is rare), the value it returns will be passed as a third “snapshot” parameter to </span>`componentDidUpdate()`<span offset-key="37182f5b86c9421e8efc3fbde97bf431:4">. Otherwise this parameter will be undefined.</span></span>

> <span key="701aecd5bb94427bb74f1a57b60fb37a"><span offset-key="701aecd5bb94427bb74f1a57b60fb37a:0">NotecomponentDidUpdate() will not be invoked if shouldComponentUpdate() returns false.</span></span>

<span key="4179bcbce4864ffe9441dc243bd559a6">`componentWillUnmount()`</span>

<span key="b6f00fb627c144d4b19fae2c8321b160">`componentWillUnmount()`</span>

<span key="d07bb5bf7e6e418686a2c759d60f7e24">`componentWillUnmount()`<span offset-key="d07bb5bf7e6e418686a2c759d60f7e24:1"> is invoked immediately before a component is unmounted and destroyed. Perform any necessary cleanup in this method, such as invalidating timers, canceling network requests, or cleaning up any subscriptions that were created in </span>`componentDidMount()`<span offset-key="d07bb5bf7e6e418686a2c759d60f7e24:3">.</span></span>

<span key="63f8042fe64f4913b706f62f49096c1b"><span offset-key="63f8042fe64f4913b706f62f49096c1b:0">You </span>**should not call** `setState()`<span offset-key="63f8042fe64f4913b706f62f49096c1b:3"> in </span>`componentWillUnmount()`<span offset-key="63f8042fe64f4913b706f62f49096c1b:5"> because the component will never be re-rendered. Once a component instance is unmounted, it will never be mounted again.</span></span>

<span key="b7b4187573ef481781918152f0aa5c23"><span offset-key="b7b4187573ef481781918152f0aa5c23:0">Rarely Used Lifecycle Methods</span></span>

<span key="dbce2680117f4e1d9e073b3372af5ca1"><span offset-key="dbce2680117f4e1d9e073b3372af5ca1:0">The methods in this section correspond to uncommon use cases. They’re handy once in a while, but most of your components probably don’t need any of them. </span>**You can see most of the methods below on** </span><a href="https://projects.wojtekmaj.pl/react-lifecycle-methods-diagram/" class="css-4rbku5 css-1dbjc4n r-1loqt21 r-1471scf r-1otgn73 r-1i6wzkk r-lrvibr"><span data-key="f24ba1b76d2945499889cca1217b4e0a" data-rnw-int-class="nearest_257-26400_259-26401-237__"><span data-key="2beff11a416542ad87e751a3e78a0b8a"><strong>this lifecycle diagram</strong></span></span></a><span key="0e5cb2b55125402fa92adf76b3708251"> **if you click the “Show less common lifecycles” checkbox at the top of it.**</span>

# 

<span key="8a5884c42cbe4ff9b7a3afbb53aa6a5d">`shouldComponentUpdate()`</span>

<span key="ad2ab0b0b5454efbaeab4909b508b178">`shouldComponentUpdate(nextProps, nextState)`</span>

<span key="65c6d26a0698444c9fc7da7dbc8abf0c"><span offset-key="65c6d26a0698444c9fc7da7dbc8abf0c:0">Use </span>`shouldComponentUpdate()`<span offset-key="65c6d26a0698444c9fc7da7dbc8abf0c:2"> to let React know if a component’s output is not affected by the current change in state or props. The default behavior is to re-render on every state change, and in the vast majority of cases you should rely on the default behavior.</span></span>

<span key="7db9b829b7c14964855e943fb8f09465">`shouldComponentUpdate()`<span offset-key="7db9b829b7c14964855e943fb8f09465:1"> is invoked before rendering when new props or state are being received. Defaults to </span>`true`<span offset-key="7db9b829b7c14964855e943fb8f09465:3">. This method is not called for the initial render or when </span>`forceUpdate()`<span offset-key="7db9b829b7c14964855e943fb8f09465:5"> is used.</span></span>

<span key="00c916b4ed47446499270fa792923b83"><span offset-key="00c916b4ed47446499270fa792923b83:0">This method only exists as a </span></span><a href="https://reactjs.org/docs/optimizing-performance.html" class="css-4rbku5 css-1dbjc4n r-1loqt21 r-1471scf r-1otgn73 r-1i6wzkk r-lrvibr"><span data-key="c14d29bc8b834d27859c04d4607d2934" data-rnw-int-class="nearest_257-26400_259-26401-237__"><span data-key="785b3943bc47472993574885feb808aa"><strong>performance optimization</strong></span></span></a><span key="7865a1f562224441b89e04a37fb7c9a9">**.**<span offset-key="7865a1f562224441b89e04a37fb7c9a9:1"> Do not rely on it to “prevent” a rendering, as this can lead to bugs. </span>**Consider using the built-in** `[PureComponent](<https://reactjs.org/docs/react-api.html#reactpurecomponent>)`<span offset-key="7865a1f562224441b89e04a37fb7c9a9:4"> instead of writing </span>`shouldComponentUpdate()`<span offset-key="7865a1f562224441b89e04a37fb7c9a9:6"> by hand. </span>`PureComponent`<span offset-key="7865a1f562224441b89e04a37fb7c9a9:8"> performs a shallow comparison of props and state, and reduces the chance that you’ll skip a necessary update.</span></span>

<span key="e7ee32f09a0f4743babbb5e75526f849"><span offset-key="e7ee32f09a0f4743babbb5e75526f849:0">If you are confident you want to write it by hand, you may compare </span>`this.props`<span offset-key="e7ee32f09a0f4743babbb5e75526f849:2"> with </span>`nextProps`<span offset-key="e7ee32f09a0f4743babbb5e75526f849:4"> and </span>`this.state`<span offset-key="e7ee32f09a0f4743babbb5e75526f849:6"> with </span>`nextState`<span offset-key="e7ee32f09a0f4743babbb5e75526f849:8"> and return </span>`false`<span offset-key="e7ee32f09a0f4743babbb5e75526f849:10"> to tell React the update can be skipped. Note that returning </span>`false`<span offset-key="e7ee32f09a0f4743babbb5e75526f849:12"> does not prevent child components from re-rendering when </span>*their*<span offset-key="e7ee32f09a0f4743babbb5e75526f849:14"> state changes.</span></span>

<span key="ccf77088626f48218ac4218afffc0ce8"><span offset-key="ccf77088626f48218ac4218afffc0ce8:0">We do not recommend doing deep equality checks or using </span>`JSON.stringify()`<span offset-key="ccf77088626f48218ac4218afffc0ce8:2"> in </span>`shouldComponentUpdate()`<span offset-key="ccf77088626f48218ac4218afffc0ce8:4">. It is very inefficient and will harm performance.</span></span>

<span key="4a8aad652d804cb088e10e8d16a04aac"><span offset-key="4a8aad652d804cb088e10e8d16a04aac:0">Currently, if </span>`shouldComponentUpdate()`<span offset-key="4a8aad652d804cb088e10e8d16a04aac:2"> returns </span>`false`<span offset-key="4a8aad652d804cb088e10e8d16a04aac:4">, then </span>`[UNSAFE_componentWillUpdate()](<https://reactjs.org/docs/react-component.html#unsafe_componentwillupdate>)`<span offset-key="4a8aad652d804cb088e10e8d16a04aac:6">, </span>`[render()](<https://reactjs.org/docs/react-component.html#render>)`<span offset-key="4a8aad652d804cb088e10e8d16a04aac:8">, and </span>`[componentDidUpdate()](<https://reactjs.org/docs/react-component.html#componentdidupdate>)`<span offset-key="4a8aad652d804cb088e10e8d16a04aac:10"> will not be invoked. In the future React may treat </span>`shouldComponentUpdate()`<span offset-key="4a8aad652d804cb088e10e8d16a04aac:12"> as a hint rather than a strict directive, and returning </span>`false`<span offset-key="4a8aad652d804cb088e10e8d16a04aac:14"> may still result in a re-rendering of the component.</span></span>

<span key="a354ed5c16e044ccb344bc628e9ea711">`static getDerivedStateFromProps()`</span>

<span key="bad367b3e46b447b81280e2e55132e44">`static getDerivedStateFromProps(props, state)`</span>

<span key="6d70752f0b85424d86744f7428213da7">`getDerivedStateFromProps`<span offset-key="6d70752f0b85424d86744f7428213da7:1"> is invoked right before calling the render method, both on the initial mount and on subsequent updates. It should return an object to update the state, or </span>`null`<span offset-key="6d70752f0b85424d86744f7428213da7:3"> to update nothing.</span></span>

<span key="01cea398ee4d4438b3330eb5fc082ccc"><span offset-key="01cea398ee4d4438b3330eb5fc082ccc:0">This method exists for </span></span><a href="https://reactjs.org/blog/2018/06/07/you-probably-dont-need-derived-state.html#when-to-use-derived-state" class="css-4rbku5 css-1dbjc4n r-1loqt21 r-1471scf r-1otgn73 r-1i6wzkk r-lrvibr"><span data-key="00e0ac34e2ca4d63a86d6f8900d0ca6b" data-rnw-int-class="nearest_257-26400_259-26401-237__"><span data-key="a6f25661efbe4e5fb117daae99c7989d"><span data-offset-key="a6f25661efbe4e5fb117daae99c7989d:0">rare use cases</span></span></span></a><span key="0ec59546211742e6aca95ae0c0271b4b"><span offset-key="0ec59546211742e6aca95ae0c0271b4b:0"> where the state depends on changes in props over time. For example, it might be handy for implementing a </span>`<Transition>`<span offset-key="0ec59546211742e6aca95ae0c0271b4b:2"> component that compares its previous and next children to decide which of them to animate in and out.</span></span>

<span key="42a2670fbfe84175a3d58f0844a05a41"><span offset-key="42a2670fbfe84175a3d58f0844a05a41:0">Deriving state leads to verbose code and makes your components difficult to think about. </span></span><a href="https://reactjs.org/blog/2018/06/07/you-probably-dont-need-derived-state.html" class="css-4rbku5 css-1dbjc4n r-1loqt21 r-1471scf r-1otgn73 r-1i6wzkk r-lrvibr"><span data-key="c4f8aa092d664daf9ffa1b5c43e9b092" data-rnw-int-class="nearest_257-26400_259-26401-237__"><span data-key="52eb050436074c48a542e3a84576c898"><span data-offset-key="52eb050436074c48a542e3a84576c898:0">Make sure you’re familiar with simpler alternatives:</span></span></span></a><span key="01eede9da342440e83d43bdca37389d5"><span offset-key="01eede9da342440e83d43bdca37389d5:0"><span slate-zero-width="z">​</span></span></span>

<span key="c110ebe25c414db68fb2e559a770b094">`getSnapshotBeforeUpdate()`</span>

<span key="dcf2b4ce6f8944dfaae19cd6b2be85ad">`getSnapshotBeforeUpdate(prevProps, prevState)`</span>

<span key="9a95d11f4d4642deab0c00109ffa0d7d">`getSnapshotBeforeUpdate()`<span offset-key="9a95d11f4d4642deab0c00109ffa0d7d:1"> is invoked right before the most recently rendered output is committed to e.g. the DOM. It enables your component to capture some information from the DOM (e.g. scroll position) before it is potentially changed. Any value returned by this lifecycle method will be passed as a parameter to </span>`componentDidUpdate()`<span offset-key="9a95d11f4d4642deab0c00109ffa0d7d:3">.</span></span>

<span key="5fa086a9626849b188b1d88a1a08bc35"><span offset-key="5fa086a9626849b188b1d88a1a08bc35:0">This use case is not common, but it may occur in UIs like a chat thread that need to handle scroll position in a special way.</span></span>

<span key="793cf40de6374091b93b0796ea0dc5b6"><span offset-key="793cf40de6374091b93b0796ea0dc5b6:0">A snapshot value (or </span>`null`<span offset-key="793cf40de6374091b93b0796ea0dc5b6:2">) should be returned.</span></span>

<span key="f58577db256e4844acc98f9110d2ed01"><span offset-key="f58577db256e4844acc98f9110d2ed01:0">For example:</span></span>

<span key="dc58841a2cea4396b1055a625e6ed50e"><span offset-key="dc58841a2cea4396b1055a625e6ed50e:0">\`class ScrollingList extends React.Component { constructor(props) { super(props); this.listRef = React.createRef(); }</span></span>

<span key="0294842a82f24d509848f60c53a63f02"><span offset-key="0294842a82f24d509848f60c53a63f02:0">getSnapshotBeforeUpdate(prevProps, prevState) { // Are we adding new items to the list? // Capture the scroll position so we can adjust scroll later. if (prevProps.list.length &lt; this.props.list.length) { const list = this.listRef.current; return list.scrollHeight - list.scrollTop; } return null; }</span></span>

<span key="62a584ef4eff4db6acb6243b1f88b189"><span offset-key="62a584ef4eff4db6acb6243b1f88b189:0">componentDidUpdate(prevProps, prevState, snapshot) { // If we have a snapshot value, we've just added new items. // Adjust scroll so these new items don't push the old ones out of view. // (snapshot here is the value returned from getSnapshotBeforeUpdate) if (snapshot !== null) { const list = this.listRef.current; list.scrollTop = list.scrollHeight - snapshot; } }</span></span>

<span key="4059cb1e76204d8d989ca243a6500f4b"><span offset-key="4059cb1e76204d8d989ca243a6500f4b:0">render() { return ( &lt;div ref={this.listRef}>{/\* ...contents... \*/}&lt;/div>); } }\`</span></span>

<span key="c25c6a04b9194952bfe70e8bf5c455c0"><span offset-key="c25c6a04b9194952bfe70e8bf5c455c0:0">In the above examples, it is important to read the </span>`scrollHeight`<span offset-key="c25c6a04b9194952bfe70e8bf5c455c0:2"> property in </span>`getSnapshotBeforeUpdate`<span offset-key="c25c6a04b9194952bfe70e8bf5c455c0:4"> because there may be delays between “render” phase lifecycles (like </span>`render`<span offset-key="c25c6a04b9194952bfe70e8bf5c455c0:6">) and “commit” phase lifecycles (like </span>`getSnapshotBeforeUpdate`<span offset-key="c25c6a04b9194952bfe70e8bf5c455c0:8"> and </span>`componentDidUpdate`<span offset-key="c25c6a04b9194952bfe70e8bf5c455c0:10">).</span></span>

<span key="ba6d8ac95b004ba6b2c67a1918a88979"><span offset-key="ba6d8ac95b004ba6b2c67a1918a88979:0">Error boundaries</span></span>

<span key="84c85c5c3dbb4dfc9287b75ddffd3da1"><span offset-key="84c85c5c3dbb4dfc9287b75ddffd3da1:0"><span slate-zero-width="z">​</span></span></span><a href="https://reactjs.org/docs/error-boundaries.html" class="css-4rbku5 css-1dbjc4n r-1loqt21 r-1471scf r-1otgn73 r-1i6wzkk r-lrvibr"><span data-key="aeef8cb8e1744dd7a94f9fe0ebf47fb6" data-rnw-int-class="nearest_257-26400_259-26401-237__"><span data-key="3dcebf844b1d4246be285904a760b777"><span data-offset-key="3dcebf844b1d4246be285904a760b777:0">Error boundaries</span></span></span></a><span key="d99fa33da3674ec0bd08c9139c5c441c"><span offset-key="d99fa33da3674ec0bd08c9139c5c441c:0"> are React components that catch JavaScript errors anywhere in their child component tree, log those errors, and display a fallback UI instead of the component tree that crashed. Error boundaries catch errors during rendering, in lifecycle methods, and in constructors of the whole tree below them.</span></span>

<span key="a70424b113994cf0adc63f73555c6525"><span offset-key="a70424b113994cf0adc63f73555c6525:0">A class component becomes an error boundary if it defines either (or both) of the lifecycle methods </span>`static getDerivedStateFromError()`<span offset-key="a70424b113994cf0adc63f73555c6525:2"> or </span>`componentDidCatch()`<span offset-key="a70424b113994cf0adc63f73555c6525:4">. Updating state from these lifecycles lets you capture an unhandled JavaScript error in the below tree and display a fallback UI.</span></span>

<span key="d00cced949cc49bc94a5442aeb7ee9b2"><span offset-key="d00cced949cc49bc94a5442aeb7ee9b2:0">Only use error boundaries for recovering from unexpected exceptions; </span>**don’t try to use them for control flow.**</span>

<span key="709791d3210e4d75b92689169c1cd726"><span offset-key="709791d3210e4d75b92689169c1cd726:0">For more details, see </span></span><a href="https://reactjs.org/blog/2017/07/26/error-handling-in-react-16.html" class="css-4rbku5 css-1dbjc4n r-1loqt21 r-1471scf r-1otgn73 r-1i6wzkk r-lrvibr"><span data-key="fcdf271820794a6e8fe9fdbb1af0e8d6" data-rnw-int-class="nearest_257-26400_259-26401-237__"><span data-key="23143aa9542b4aaa90fbece4eefd2999"><em>Error Handling in React 16</em></span></span></a><span key="72b3e89039b64bc393ff1552ef7e401d"><span offset-key="72b3e89039b64bc393ff1552ef7e401d:0">.</span></span>

> <span key="b57c9382f1ff481781b9f4e926893bad"><span offset-key="b57c9382f1ff481781b9f4e926893bad:0">NoteError boundaries only catch errors in the components below them in the tree. An error boundary can’t catch an error within itself.</span></span>

<span key="38bfb09b61d540da87224f531f039111">`static getDerivedStateFromError()`</span>

<span key="ac44e3c0b08147748132a128c68f53d0">`static getDerivedStateFromError(error)`</span>

<span key="6a63d16471d648ca82d17f7ffaa6385b">`setState()`</span>

<span key="68b97048babe4db9be24548d81c43758">`setState(updater, [callback])`</span>

<span key="7ff84aa4d8464bf7a543134c2430a20a">`setState()`<span offset-key="7ff84aa4d8464bf7a543134c2430a20a:1"> enqueues changes to the component state and tells React that this component and its children need to be re-rendered with the updated state. This is the primary method you use to update the user interface in response to event handlers and server responses.</span></span>

<span key="5699f81522ca4b2984dc5c6d0ea13336"><span offset-key="5699f81522ca4b2984dc5c6d0ea13336:0">Think of </span>`setState()`<span offset-key="5699f81522ca4b2984dc5c6d0ea13336:2"> as a </span>*request*<span offset-key="5699f81522ca4b2984dc5c6d0ea13336:4"> rather than an immediate command to update the component. For better perceived performance, React may delay it, and then update several components in a single pass. React does not guarantee that the state changes are applied immediately.</span></span>

<span key="d4364a3cdfe0419aa8beaaf7be7a4489">`setState()`<span offset-key="d4364a3cdfe0419aa8beaaf7be7a4489:1"> does not always immediately update the component. It may batch or defer the update until later. This makes reading </span>`this.state`<span offset-key="d4364a3cdfe0419aa8beaaf7be7a4489:3"> right after calling </span>`setState()`<span offset-key="d4364a3cdfe0419aa8beaaf7be7a4489:5"> a potential pitfall. Instead, use </span>`componentDidUpdate`<span offset-key="d4364a3cdfe0419aa8beaaf7be7a4489:7"> or a </span>`setState`<span offset-key="d4364a3cdfe0419aa8beaaf7be7a4489:9"> callback (</span>`setState(updater, callback)`<span offset-key="d4364a3cdfe0419aa8beaaf7be7a4489:11">), either of which are guaranteed to fire after the update has been applied. If you need to set the state based on the previous state, read about the </span>`updater`<span offset-key="d4364a3cdfe0419aa8beaaf7be7a4489:13"> argument below.</span></span>

<span key="198f6924d03c4b0da075bcbef44a86c6">`setState()`<span offset-key="198f6924d03c4b0da075bcbef44a86c6:1"> will always lead to a re-render unless </span>`shouldComponentUpdate()`<span offset-key="198f6924d03c4b0da075bcbef44a86c6:3"> returns </span>`false`<span offset-key="198f6924d03c4b0da075bcbef44a86c6:5">. If mutable objects are being used and conditional rendering logic cannot be implemented in </span>`shouldComponentUpdate()`<span offset-key="198f6924d03c4b0da075bcbef44a86c6:7">, calling </span>`setState()`<span offset-key="198f6924d03c4b0da075bcbef44a86c6:9"> only when the new state differs from the previous state will avoid unnecessary re-renders.</span></span>

<span key="9e0f06426bc8480ab3b8d3a3df036124"><span offset-key="9e0f06426bc8480ab3b8d3a3df036124:0">The first argument is an </span>`updater`<span offset-key="9e0f06426bc8480ab3b8d3a3df036124:2"> function with the signature:</span></span>

<span key="1ecdc50a97b84843b8c7beac8352e31d">`(state, props) => stateChange`</span>

<span key="afefcc91c841473b9cacb532298acf78">`state`<span offset-key="afefcc91c841473b9cacb532298acf78:1"> is a reference to the component state at the time the change is being applied. It should not be directly mutated. Instead, changes should be represented by building a new object based on the input from </span>`state`<span offset-key="afefcc91c841473b9cacb532298acf78:3"> and </span>`props`<span offset-key="afefcc91c841473b9cacb532298acf78:5">. <span key="00f0b2275d834b669270504cefad025d">`this.setState((state, props) => { return {counter: state.counter + props.step}; });`</span>

<span key="06b2593c7b14422988898cb454d96aa7"><span offset-key="06b2593c7b14422988898cb454d96aa7:0">Both </span>`state`<span offset-key="06b2593c7b14422988898cb454d96aa7:2"> and </span>`props`<span offset-key="06b2593c7b14422988898cb454d96aa7:4"> received by the updater function are guaranteed to be up-to-date. <span key="f88b763e0aed4550815e5b46ad2d2ad7"><span offset-key="f88b763e0aed4550815e5b46ad2d2ad7:0">The second parameter to </span>`setState()`<span offset-key="f88b763e0aed4550815e5b46ad2d2ad7:2"> is an optional callback function that will be executed once </span>`setState`<span offset-key="f88b763e0aed4550815e5b46ad2d2ad7:4"> is completed and the component is re-rendered. <span key="7ca296a9cc3e4006a2aa7f11eb129828"><span offset-key="7ca296a9cc3e4006a2aa7f11eb129828:0">You may optionally pass an object as the first argument to </span>`setState()`<span offset-key="7ca296a9cc3e4006a2aa7f11eb129828:2"> instead of a function:</span></span>

<span key="e328c65a40384f1a8df10aa4fd3152ae">`this.setState({quantity: 2})`</span>

<span key="31ea5a7cd8b54a90960af8f8ed637d66"><span offset-key="31ea5a7cd8b54a90960af8f8ed637d66:0">This form of </span>`setState()`<span offset-key="31ea5a7cd8b54a90960af8f8ed637d66:2"> is also asynchronous, and multiple calls during the same cycle may be batched together. For example, if you attempt to increment an item quantity more than once in the same cycle, that will result in the equivalent of:</span></span>

<span key="50cf35fb2517456f9781d595007810aa">`Object.assign( previousState, {quantity: state.quantity + 1}, {quantity: state.quantity + 1}, ... )`</span>

<span key="6ca2c2be3e634b66a999d92f6fdce86b"><span offset-key="6ca2c2be3e634b66a999d92f6fdce86b:0">Subsequent calls will override values from previous calls in the same cycle, so the quantity will only be incremented once. If the next state depends on the current state, we recommend using the updater function form, instead:</span></span>

<span key="b11c555adf4246f98bd973df66ea0895">`this.setState((state) => { return {quantity: state.quantity + 1}; });`</span>

<span key="13f6efd411c245569135952ce4e9e21f">`forceUpdate()`</span>

<span key="c29a988edeca4e6281013a3e970f24ec">`component.forceUpdate(callback)`</span>

<span key="dd33330d942b43c2be20717c722720eb"><span offset-key="dd33330d942b43c2be20717c722720eb:0">By default, when your component’s state or props change, your component will re-render. If your </span>`render()`<span offset-key="dd33330d942b43c2be20717c722720eb:2"> method depends on some other data, you can tell React that the component needs re-rendering by calling </span>`forceUpdate()`<span offset-key="dd33330d942b43c2be20717c722720eb:4">.</span></span>

<span key="af8b62adf0604d29b04495e38c922438"><span offset-key="af8b62adf0604d29b04495e38c922438:0">Calling </span>`forceUpdate()`<span offset-key="af8b62adf0604d29b04495e38c922438:2"> will cause </span>`render()`<span offset-key="af8b62adf0604d29b04495e38c922438:4"> to be called on the component, skipping </span>`shouldComponentUpdate()`<span offset-key="af8b62adf0604d29b04495e38c922438:6">. This will trigger the normal lifecycle methods for child components, including the </span>`shouldComponentUpdate()`<span offset-key="af8b62adf0604d29b04495e38c922438:8"> method of each child. React will still only update the DOM if the markup changes.</span></span>

<span key="3edad14142ef4aa78d832c114fb305db"><span offset-key="3edad14142ef4aa78d832c114fb305db:0">Normally you should try to avoid all uses of </span>`forceUpdate()`<span offset-key="3edad14142ef4aa78d832c114fb305db:2"> and only read from </span>`this.props`<span offset-key="3edad14142ef4aa78d832c114fb305db:4"> and </span>`this.state`<span offset-key="3edad14142ef4aa78d832c114fb305db:6"> in </span>`render()`<span offset-key="3edad14142ef4aa78d832c114fb305db:8">.</span></span>
