<span key="3effc8c19b3449b6b78215feb2b81d88"><span offset-key="3effc8c19b3449b6b78215feb2b81d88:0">Blockchain Basics</span></span>

<span key="af3103eb04b841078dda9c4df50672e0"><span offset-key="af3103eb04b841078dda9c4df50672e0:0">Blockchains as a concept are not too hard to understand for programmers. The reason is that most of the complications (mining, </span></span><a href="https://en.wikipedia.org/wiki/Cryptographic_hash_function" class="css-4rbku5 css-1dbjc4n r-1loqt21 r-1471scf r-1otgn73 r-1i6wzkk r-lrvibr"><span data-key="9453db0416e64a4d8caa79f9bd1b960c" data-rnw-int-class="nearest_257-5835_259-5836-237__"><span data-key="b3c42d87e4b549248f30333d333e4d39"><span data-offset-key="b3c42d87e4b549248f30333d333e4d39:0">hashing</span></span></span></a><span key="ead77755d51646fbb61b5bf637c85ca8"><span offset-key="ead77755d51646fbb61b5bf637c85ca8:0">, </span></span><a href="https://en.wikipedia.org/wiki/Elliptic_curve_cryptography" class="css-4rbku5 css-1dbjc4n r-1loqt21 r-1471scf r-1otgn73 r-1i6wzkk r-lrvibr"><span data-key="500d25d2ed7a4b7c9b1bd6b7b206d3be" data-rnw-int-class="nearest_257-5835_259-5836-237__"><span data-key="77e4da7ce0404a3384e94b75303f4357"><span data-offset-key="77e4da7ce0404a3384e94b75303f4357:0">elliptic-curve cryptography</span></span></span></a><span key="4990988ccd074187b4118e22ada80116"><span offset-key="4990988ccd074187b4118e22ada80116:0">, </span></span><a href="https://en.wikipedia.org/wiki/Peer-to-peer" class="css-4rbku5 css-1dbjc4n r-1loqt21 r-1471scf r-1otgn73 r-1i6wzkk r-lrvibr"><span data-key="64dbb8f23fdc4eda81723d4a765a682a" data-rnw-int-class="nearest_257-5835_259-5836-237__"><span data-key="90976bacb8ff4580a165b2db929c7246"><span data-offset-key="90976bacb8ff4580a165b2db929c7246:0">peer-to-peer networks</span></span></span></a><span key="6141f8dcc9b449878e761a0c283caa46"><span offset-key="6141f8dcc9b449878e761a0c283caa46:0">, etc.) are just there to provide a certain set of features and promises for the platform. Once you accept these features as given, you do not have to worry about the underlying technology - or do you have to know how Amazon’s AWS works internally in order to use it?</span></span>

### 

<span key="324655865aaa4c4abbb60fcac5c942a6"><span offset-key="324655865aaa4c4abbb60fcac5c942a6:0">Transactions</span></span>

<span key="02cb53abc5b6443b906856e3f070e6fa"><span offset-key="02cb53abc5b6443b906856e3f070e6fa:0">A blockchain is a globally shared, transactional database. This means that everyone can read entries in the database just by participating in the network. If you want to change something in the database, you have to create a so-called transaction which has to be accepted by all others. The word transaction implies that the change you want to make (assume you want to change two values at the same time) is either not done at all or completely applied. Furthermore, while your transaction is being applied to the database, no other transaction can alter it.</span></span>

<span key="a984d002b78d4e3bb598f5f1d59602e9"><span offset-key="a984d002b78d4e3bb598f5f1d59602e9:0">As an example, imagine a table that lists the balances of all accounts in an electronic currency. If a transfer from one account to another is requested, the transactional nature of the database ensures that if the amount is subtracted from one account, it is always added to the other account. If due to whatever reason, adding the amount to the target account is not possible, the source account is also not modified.</span></span>

<span key="6659920a7ce0498da75cdde43d6283d2"><span offset-key="6659920a7ce0498da75cdde43d6283d2:0">Furthermore, a transaction is always cryptographically signed by the sender (creator). This makes it straightforward to guard access to specific modifications of the database. In the example of the electronic currency, a simple check ensures that only the person holding the keys to the account can transfer money from it.</span></span>

### 

<span key="54e3012eb6294982b2499fbb02cf525c"><span offset-key="54e3012eb6294982b2499fbb02cf525c:0">Blocks</span></span>

<span key="7118e974fb9644c3bcdd25615318c4c5"><span offset-key="7118e974fb9644c3bcdd25615318c4c5:0">One major obstacle to overcome is what (in Bitcoin terms) is called a “double-spend attack”: What happens if two transactions exist in the network that both want to empty an account? Only one of the transactions can be valid, typically the one that is accepted first. The problem is that “first” is not an objective term in a peer-to-peer network.</span></span>

<span key="59cbde26131d40ab8f0b21b4a2b653bc"><span offset-key="59cbde26131d40ab8f0b21b4a2b653bc:0">The abstract answer to this is that you do not have to care. A globally accepted order of the transactions will be selected for you, solving the conflict. The transactions will be bundled into what is called a “block” and then they will be executed and distributed among all participating nodes. If two transactions contradict each other, the one that ends up being second will be rejected and not become part of the block.</span></span>

<span key="539d8c05b1ff4a95832c098600e6c660"><span offset-key="539d8c05b1ff4a95832c098600e6c660:0">These blocks form a linear sequence in time and that is where the word “blockchain” derives from. Blocks are added to the chain in rather regular intervals - for Ethereum this is roughly every 17 seconds.</span></span>

<span key="e2334d14880b464f9be759683edd092f"><span offset-key="e2334d14880b464f9be759683edd092f:0">As part of the “order selection mechanism” (which is called “mining”) it may happen that blocks are reverted from time to time, but only at the “tip” of the chain. The more blocks are added on top of a particular block, the less likely this block will be reverted. So it might be that your transactions are reverted and even removed from the blockchain, but the longer you wait, the less likely it will be.</span></span>

<span key="4d725040c2904a8ea008840bdaffb0d9"><span offset-key="4d725040c2904a8ea008840bdaffb0d9:0">Note</span></span>

<span key="078ab1cafc144a7682c751cb7f8d8aaf"><span offset-key="078ab1cafc144a7682c751cb7f8d8aaf:0">Transactions are not guaranteed to be included in the next block or any specific future block, since it is not up to the submitter of a transaction, but up to the miners to determine in which block the transaction is included.</span></span>

<span key="6970760271134e01bde1287fd5b035aa"><span offset-key="6970760271134e01bde1287fd5b035aa:0">If you want to schedule future calls of your contract, you can use the </span></span><a href="https://www.ethereum-alarm-clock.com" class="css-4rbku5 css-1dbjc4n r-1loqt21 r-1471scf r-1otgn73 r-1i6wzkk r-lrvibr"><span data-key="3141dfe02de34818a269adba84460202" data-rnw-int-class="nearest_257-5835_259-5836-237__"><span data-key="b2d02b6952724c87a2f18777b9c360a1"><span data-offset-key="b2d02b6952724c87a2f18777b9c360a1:0">alarm clock</span></span></span></a><span key="36007639c2d24880ad6faf038bdb5987"><span offset-key="36007639c2d24880ad6faf038bdb5987:0"> or a similar oracle service.</span></span>

## 

<span key="5cd7c76e93054932bda3bbf80b70ac3e"><span offset-key="5cd7c76e93054932bda3bbf80b70ac3e:0">The Ethereum Virtual Machine</span></span>

### 

<span key="e05cd02f2b00445cb463ecd2f16d18ae"><span offset-key="e05cd02f2b00445cb463ecd2f16d18ae:0">Overview</span></span>

<span key="d43c316be9dd44c9b6d2d722c4696a7e"><span offset-key="d43c316be9dd44c9b6d2d722c4696a7e:0">The Ethereum Virtual Machine or EVM is the runtime environment for smart contracts in Ethereum. It is not only sandboxed but actually completely isolated, which means that code running inside the EVM has no access to network, filesystem or other processes. Smart contracts even have limited access to other smart contracts.</span></span>

### 

<span key="f80ebd3d00054f5193eba4db649ab27f"><span offset-key="f80ebd3d00054f5193eba4db649ab27f:0">Accounts</span></span>

<span key="c114d0107a9b408e969275599c11846b"><span offset-key="c114d0107a9b408e969275599c11846b:0">There are two kinds of accounts in Ethereum which share the same address space: </span>**External accounts**<span offset-key="c114d0107a9b408e969275599c11846b:2"> that are controlled by public-private key pairs (i.e. humans) and </span>**contract accounts**<span offset-key="c114d0107a9b408e969275599c11846b:4"> which are controlled by the code stored together with the account.</span></span>

<span key="1555c7e3f4b14234bb4891e81926cd9a"><span offset-key="1555c7e3f4b14234bb4891e81926cd9a:0">The address of an external account is determined from the public key while the address of a contract is determined at the time the contract is created (it is derived from the creator address and the number of transactions sent from that address, the so-called “nonce”).</span></span>

<span key="3540c08fc3b6425a8646c177178619ba"><span offset-key="3540c08fc3b6425a8646c177178619ba:0">Regardless of whether or not the account stores code, the two types are treated equally by the EVM.</span></span>

<span key="8305272737f14db98e51cc4282c8e0e6"><span offset-key="8305272737f14db98e51cc4282c8e0e6:0">Every account has a persistent key-value store mapping 256-bit words to 256-bit words called </span>**storage**<span offset-key="8305272737f14db98e51cc4282c8e0e6:2">.</span></span>

<span key="763733d47ef54272903c17ddaaf26789"><span offset-key="763733d47ef54272903c17ddaaf26789:0">Furthermore, every account has a </span>**balance**<span offset-key="763733d47ef54272903c17ddaaf26789:2"> in Ether (in “Wei” to be exact, </span>`1 ether`<span offset-key="763733d47ef54272903c17ddaaf26789:4"> is </span>`10**18 wei`<span offset-key="763733d47ef54272903c17ddaaf26789:6">) which can be modified by sending transactions that include Ether.</span></span>

### 

<span key="9f1492b6f93648c198bc646c0c67ab33"><span offset-key="9f1492b6f93648c198bc646c0c67ab33:0">Transactions</span></span>

<span key="9e554df9315448aeaeb3caf0690b175d"><span offset-key="9e554df9315448aeaeb3caf0690b175d:0">A transaction is a message that is sent from one account to another account (which might be the same or empty, see below). It can include binary data (which is called “payload”) and Ether.</span></span>

<span key="d128cb86591342bba85b3892959fa522"><span offset-key="d128cb86591342bba85b3892959fa522:0">If the target account contains code, that code is executed and the payload is provided as input data.</span></span>

<span key="343e270cceb44605a4f3c1b3d34e5661"><span offset-key="343e270cceb44605a4f3c1b3d34e5661:0">If the target account is not set (the transaction does not have a recipient or the recipient is set to </span>`null`<span offset-key="343e270cceb44605a4f3c1b3d34e5661:2">), the transaction creates a </span>**new contract**<span offset-key="343e270cceb44605a4f3c1b3d34e5661:4">. As already mentioned, the address of that contract is not the zero address but an address derived from the sender and its number of transactions sent (the “nonce”). The payload of such a contract creation transaction is taken to be EVM bytecode and executed. The output data of this execution is permanently stored as the code of the contract. This means that in order to create a contract, you do not send the actual code of the contract, but in fact code that returns that code when executed.</span></span>

<span key="bb71e6beb61849feb23c98d1c36560eb"><span offset-key="bb71e6beb61849feb23c98d1c36560eb:0">Note</span></span>

<span key="629a37de5e8e4d3cbebd46bcb85d87e9"><span offset-key="629a37de5e8e4d3cbebd46bcb85d87e9:0">While a contract is being created, its code is still empty. Because of that, you should not call back into the contract under construction until its constructor has finished executing.</span></span>

### 

<span key="71a5823a2b05472884638f357eb3dacf"><span offset-key="71a5823a2b05472884638f357eb3dacf:0">Gas</span></span>

<span key="f43bae47d7b24f9da9154f44658b80e5"><span offset-key="f43bae47d7b24f9da9154f44658b80e5:0">Upon creation, each transaction is charged with a certain amount of </span>**gas**<span offset-key="f43bae47d7b24f9da9154f44658b80e5:2">, whose purpose is to limit the amount of work that is needed to execute the transaction and to pay for this execution at the same time. While the EVM executes the transaction, the gas is gradually depleted according to specific rules.</span></span>

<span key="7f57848f9160449bb9b26174ff351c4c"><span offset-key="7f57848f9160449bb9b26174ff351c4c:0">The </span>**gas price**<span offset-key="7f57848f9160449bb9b26174ff351c4c:2"> is a value set by the creator of the transaction, who has to pay </span>`gas_price * gas`<span offset-key="7f57848f9160449bb9b26174ff351c4c:4"> up front from the sending account. If some gas is left after the execution, it is refunded to the creator in the same way.</span></span>

<span key="36ddeebe1be04893a38151655ae09b21"><span offset-key="36ddeebe1be04893a38151655ae09b21:0">If the gas is used up at any point (i.e. it would be negative), an out-of-gas exception is triggered, which reverts all modifications made to the state in the current call frame.</span></span>

### 

<span key="f081775adc7449a58b120e9a38be9484"><span offset-key="f081775adc7449a58b120e9a38be9484:0">Storage, Memory and the Stack</span></span>

<span key="bed2275f005b4d959f7bccf9de4b1322"><span offset-key="bed2275f005b4d959f7bccf9de4b1322:0">The Ethereum Virtual Machine has three areas where it can store data- storage, memory and the stack, which are explained in the following paragraphs.</span></span>

<span key="28bf8aec3e764f2a91f486a87a3ad91f"><span offset-key="28bf8aec3e764f2a91f486a87a3ad91f:0">Each account has a data area called </span>**storage**<span offset-key="28bf8aec3e764f2a91f486a87a3ad91f:2">, which is persistent between function calls and transactions. Storage is a key-value store that maps 256-bit words to 256-bit words. It is not possible to enumerate storage from within a contract, it is comparatively costly to read, and even more to initialise and modify storage. Because of this cost, you should minimize what you store in persistent storage to what the contract needs to run. Store data like derived calculations, caching, and aggregates outside of the contract. A contract can neither read nor write to any storage apart from its own.</span></span>

<span key="572617842f3d442e9500baa16f234fe0"><span offset-key="572617842f3d442e9500baa16f234fe0:0">The second data area is called </span>**memory**<span offset-key="572617842f3d442e9500baa16f234fe0:2">, of which a contract obtains a freshly cleared instance for each message call. Memory is linear and can be addressed at byte level, but reads are limited to a width of 256 bits, while writes can be either 8 bits or 256 bits wide. Memory is expanded by a word (256-bit), when accessing (either reading or writing) a previously untouched memory word (i.e. any offset within a word). At the time of expansion, the cost in gas must be paid. Memory is more costly the larger it grows (it scales quadratically).</span></span>

<span key="704caa85c50e41c9a22f9ba2a2fefeb7"><span offset-key="704caa85c50e41c9a22f9ba2a2fefeb7:0">The EVM is not a register machine but a stack machine, so all computations are performed on a data area called the </span>**stack**<span offset-key="704caa85c50e41c9a22f9ba2a2fefeb7:2">. It has a maximum size of 1024 elements and contains words of 256 bits. Access to the stack is limited to the top end in the following way: It is possible to copy one of the topmost 16 elements to the top of the stack or swap the topmost element with one of the 16 elements below it. All other operations take the topmost two (or one, or more, depending on the operation) elements from the stack and push the result onto the stack. Of course it is possible to move stack elements to storage or memory in order to get deeper access to the stack, but it is not possible to just access arbitrary elements deeper in the stack without first removing the top of the stack.</span></span>

### 

<span key="9bc82bdf20524a5fba66040183c04fda"><span offset-key="9bc82bdf20524a5fba66040183c04fda:0">Instruction Set</span></span>

<span key="c9e7e3ca07ec422aa2d1d57b648b5413"><span offset-key="c9e7e3ca07ec422aa2d1d57b648b5413:0">The instruction set of the EVM is kept minimal in order to avoid incorrect or inconsistent implementations which could cause consensus problems. All instructions operate on the basic data type, 256-bit words or on slices of memory (or other byte arrays). The usual arithmetic, bit, logical and comparison operations are present. Conditional and unconditional jumps are possible. Furthermore, contracts can access relevant properties of the current block like its number and timestamp.</span></span>

<span key="a0fcdbbc67f04bb7ae4535380b0599a4"><span offset-key="a0fcdbbc67f04bb7ae4535380b0599a4:0">For a complete list, please see the </span></span><a href="https://docs.soliditylang.org/en/v0.8.7/yul.html#opcodes" class="css-4rbku5 css-1dbjc4n r-1loqt21 r-1471scf r-1otgn73 r-1i6wzkk r-lrvibr"><span data-key="08101364a3e940d1947c4c0602a452a2" data-rnw-int-class="nearest_257-5835_259-5836-237__"><span data-key="84aa27ed4cca4d1ca3182ce24cee3f58"><span data-offset-key="84aa27ed4cca4d1ca3182ce24cee3f58:0">list of opcodes</span></span></span></a><span key="0231508c997549c4a786841bfb38973d"><span offset-key="0231508c997549c4a786841bfb38973d:0"> as part of the inline assembly documentation.</span></span>

### 

<span key="c063f11dfd314ea184dacdda176f0882"><span offset-key="c063f11dfd314ea184dacdda176f0882:0">Message Calls</span></span>

<span key="7c8ffb5de4eb402f9af3f7c8e5fa7e05"><span offset-key="7c8ffb5de4eb402f9af3f7c8e5fa7e05:0">Contracts can call other contracts or send Ether to non-contract accounts by the means of message calls. Message calls are similar to transactions, in that they have a source, a target, data payload, Ether, gas and return data. In fact, every transaction consists of a top-level message call which in turn can create further message calls.</span></span>

<span key="9b023dbcd8c84ea988a008d3b1dcf73d"><span offset-key="9b023dbcd8c84ea988a008d3b1dcf73d:0">A contract can decide how much of its remaining </span>**gas**<span offset-key="9b023dbcd8c84ea988a008d3b1dcf73d:2"> should be sent with the inner message call and how much it wants to retain. If an out-of-gas exception happens in the inner call (or any other exception), this will be signaled by an error value put onto the stack. In this case, only the gas sent together with the call is used up. In Solidity, the calling contract causes a manual exception by default in such situations, so that exceptions “bubble up” the call stack.</span></span>

<span key="3dafa8159ba44585ae3bafa89dcb736a"><span offset-key="3dafa8159ba44585ae3bafa89dcb736a:0">As already said, the called contract (which can be the same as the caller) will receive a freshly cleared instance of memory and has access to the call payload - which will be provided in a separate area called the </span>**calldata**<span offset-key="3dafa8159ba44585ae3bafa89dcb736a:2">. After it has finished execution, it can return data which will be stored at a location in the caller’s memory preallocated by the caller. All such calls are fully synchronous.</span></span>

<span key="4b71f47ce7e94c1b8a4b38c4b7d09cfe"><span offset-key="4b71f47ce7e94c1b8a4b38c4b7d09cfe:0">Calls are </span>**limited**<span offset-key="4b71f47ce7e94c1b8a4b38c4b7d09cfe:2"> to a depth of 1024, which means that for more complex operations, loops should be preferred over recursive calls. Furthermore, only 63/64th of the gas can be forwarded in a message call, which causes a depth limit of a little less than 1000 in practice.</span></span>

### 

<span key="bb9d526d657d44029f60c32ea7e0ea26"><span offset-key="bb9d526d657d44029f60c32ea7e0ea26:0">Delegatecall / Callcode and Libraries</span></span>

<span key="1cfc9fdbb71b4bfd850e04b21f415997"><span offset-key="1cfc9fdbb71b4bfd850e04b21f415997:0">There exists a special variant of a message call, named </span>**delegatecall**<span offset-key="1cfc9fdbb71b4bfd850e04b21f415997:2"> which is identical to a message call apart from the fact that the code at the target address is executed in the context of the calling contract and </span>`msg.sender`<span offset-key="1cfc9fdbb71b4bfd850e04b21f415997:4"> and </span>`msg.value`<span offset-key="1cfc9fdbb71b4bfd850e04b21f415997:6"> do not change their values.</span></span>

<span key="31bb58cb47dd4e3082e0bcbc905663c2"><span offset-key="31bb58cb47dd4e3082e0bcbc905663c2:0">This means that a contract can dynamically load code from a different address at runtime. Storage, current address and balance still refer to the calling contract, only the code is taken from the called address.</span></span>

<span key="637ab49fc8ea4ad386125f8f4ae79ccd"><span offset-key="637ab49fc8ea4ad386125f8f4ae79ccd:0">This makes it possible to implement the “library” feature in Solidity: Reusable library code that can be applied to a contract’s storage, e.g. in order to implement a complex data structure.</span></span>

### 

<span key="ab2057f3ee704cc882f36962be7937ce"><span offset-key="ab2057f3ee704cc882f36962be7937ce:0">Logs</span></span>

<span key="d38a4927ce7a499fbb0f5f7cd7f9f262"><span offset-key="d38a4927ce7a499fbb0f5f7cd7f9f262:0">It is possible to store data in a specially indexed data structure that maps all the way up to the block level. This feature called </span>**logs**<span offset-key="d38a4927ce7a499fbb0f5f7cd7f9f262:2"> is used by Solidity in order to implement </span></span><a href="https://docs.soliditylang.org/en/v0.8.7/contracts.html#events" class="css-4rbku5 css-1dbjc4n r-1loqt21 r-1471scf r-1otgn73 r-1i6wzkk r-lrvibr"><span data-key="c63058f381644569b6437025121ca116" data-rnw-int-class="nearest_257-5835_259-5836-237__"><span data-key="53d06b5faed942d8837bb9cace58c528"><span data-offset-key="53d06b5faed942d8837bb9cace58c528:0">events</span></span></span></a><span key="a84e38109ee241ee826a6d456ce4ed8e"><span offset-key="a84e38109ee241ee826a6d456ce4ed8e:0">. Contracts cannot access log data after it has been created, but they can be efficiently accessed from outside the blockchain. Since some part of the log data is stored in </span></span><a href="https://en.wikipedia.org/wiki/Bloom_filter" class="css-4rbku5 css-1dbjc4n r-1loqt21 r-1471scf r-1otgn73 r-1i6wzkk r-lrvibr"><span data-key="7c569fd5357a4e3aa6231c6ed74ecd96" data-rnw-int-class="nearest_257-5835_259-5836-237__"><span data-key="7ac0bd4686fd451f9946828dd182490e"><span data-offset-key="7ac0bd4686fd451f9946828dd182490e:0">bloom filters</span></span></span></a><span key="ebb373a180514095b43052bd0547282f"><span offset-key="ebb373a180514095b43052bd0547282f:0">, it is possible to search for this data in an efficient and cryptographically secure way, so network peers that do not download the whole blockchain (so-called “light clients”) can still find these logs.</span></span>

### 

<span key="40dd5312e5ac49219246e95bf46259a6"><span offset-key="40dd5312e5ac49219246e95bf46259a6:0">Create</span></span>

<span key="f2df1118ed13430c816a174c64798d9e"><span offset-key="f2df1118ed13430c816a174c64798d9e:0">Contracts can even create other contracts using a special opcode (i.e. they do not simply call the zero address as a transaction would). The only difference between these </span>**create calls**<span offset-key="f2df1118ed13430c816a174c64798d9e:2"> and normal message calls is that the payload data is executed and the result stored as code and the caller / creator receives the address of the new contract on the stack.</span></span>

### 

<span key="56194c2282024a408d9db67b0970fd48"><span offset-key="56194c2282024a408d9db67b0970fd48:0">Deactivate and Self-destruct</span></span>

<span key="e1782f4aeeaf44b3b83feca455f34446"><span offset-key="e1782f4aeeaf44b3b83feca455f34446:0">The only way to remove code from the blockchain is when a contract at that address performs the </span>`selfdestruct`<span offset-key="e1782f4aeeaf44b3b83feca455f34446:2"> operation. The remaining Ether stored at that address is sent to a designated target and then the storage and code is removed from the state. Removing the contract in theory sounds like a good idea, but it is potentially dangerous, as if someone sends Ether to removed contracts, the Ether is forever lost.</span></span>

<span key="b962cea2bfa44cc9816c4f593b0e1e4d"><span offset-key="b962cea2bfa44cc9816c4f593b0e1e4d:0">Warning</span></span>

<span key="30e75d75e267444bb3b8f7f5603e9791"><span offset-key="30e75d75e267444bb3b8f7f5603e9791:0">Even if a contract is removed by </span>`selfdestruct`<span offset-key="30e75d75e267444bb3b8f7f5603e9791:2">, it is still part of the history of the blockchain and probably retained by most Ethereum nodes. So using </span>`selfdestruct`<span offset-key="30e75d75e267444bb3b8f7f5603e9791:4"> is not the same as deleting data from a hard disk.</span></span>

<span key="8833fe6b66974bf8b8aa3f599f9d4e35"><span offset-key="8833fe6b66974bf8b8aa3f599f9d4e35:0">Note</span></span>

<span key="276c85ca57d849b381d82b5c78914b84"><span offset-key="276c85ca57d849b381d82b5c78914b84:0">Even if a contract’s code does not contain a call to </span>`selfdestruct`<span offset-key="276c85ca57d849b381d82b5c78914b84:2">, it can still perform that operation using </span>`delegatecall`<span offset-key="276c85ca57d849b381d82b5c78914b84:4"> or </span>`callcode`<span offset-key="276c85ca57d849b381d82b5c78914b84:6">.</span></span>

<span key="692cf6bfaac248c29c301b86fea633ab"><span offset-key="692cf6bfaac248c29c301b86fea633ab:0">If you want to deactivate your contracts, you should instead </span>**disable**<span offset-key="692cf6bfaac248c29c301b86fea633ab:2"> them by changing some internal state which causes all functions to revert. This makes it impossible to use the contract, as it returns Ether immediately.</span></span><a href="https://docs.soliditylang.org/en/v0.8.7/installing-solidity.html" class="css-4rbku5 css-1dbjc4n r-1loqt21 r-1471scf r-1otgn73 r-1i6wzkk r-lrvibr"><span data-key="dd676e9abe834051aea2b598c9e94f88" data-rnw-int-class="nearest_257-5835_259-5836-237__"><span data-key="0f5e761ec0d04ef18b6c8dc8529ec6a4"><span data-offset-key="0f5e761ec0d04ef18b6c8dc8529ec6a4:0">Next </span></span></span></a><span key="eaa1f968260d4fb5b22adb171f059a3c"><span offset-key="eaa1f968260d4fb5b22adb171f059a3c:0"><span slate-zero-width="z">​</span></span></span><a href="https://docs.soliditylang.org/en/v0.8.7/index.html" class="css-4rbku5 css-1dbjc4n r-1loqt21 r-1471scf r-1otgn73 r-1i6wzkk r-lrvibr"><span data-key="ca2d34105f60447894c6caab0bf2ce90" data-rnw-int-class="nearest_257-5835_259-5836-237__"><span data-key="7317e2bad3c94cd586a59240b2c85ed9"><span data-offset-key="7317e2bad3c94cd586a59240b2c85ed9:0"> Previous</span></span></span></a><span key="3698478927744a15a2f4347276d347fa"><span offset-key="3698478927744a15a2f4347276d347fa:0"> </span></span>
