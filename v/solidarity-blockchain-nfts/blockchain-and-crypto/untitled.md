Blockchain Basics

## 

<span key="7545b07df35f46a7939882a07e35c539"><span offset-key="7545b07df35f46a7939882a07e35c539:0">Blockchain Basics</span></span>

<span key="50cf544219714a0db9b3028d03872be6"><span offset-key="50cf544219714a0db9b3028d03872be6:0">Blockchains as a concept are not too hard to understand for programmers. The reason is that most of the complications (mining, </span></span><a href="https://en.wikipedia.org/wiki/Cryptographic_hash_function" class="css-4rbku5 css-1dbjc4n r-1loqt21 r-1471scf r-1otgn73 r-1i6wzkk r-lrvibr"><span data-key="cfd9571a85084a6fa3b32052106a7a89" data-rnw-int-class="nearest_257-6651_259-6652-237__"><span data-key="c0896a56ea3a428ba54122cbea8db7b3"><span data-offset-key="c0896a56ea3a428ba54122cbea8db7b3:0">hashing</span></span></span></a><span key="7ce7c23e59e24d7dacd974660dc6be70"><span offset-key="7ce7c23e59e24d7dacd974660dc6be70:0">, </span></span><a href="https://en.wikipedia.org/wiki/Elliptic_curve_cryptography" class="css-4rbku5 css-1dbjc4n r-1loqt21 r-1471scf r-1otgn73 r-1i6wzkk r-lrvibr"><span data-key="a3507da51a8a449c9757872d2538b915" data-rnw-int-class="nearest_257-6651_259-6652-237__"><span data-key="a84db6d8473c4e979b7f6a669f7f94de"><span data-offset-key="a84db6d8473c4e979b7f6a669f7f94de:0">elliptic-curve cryptography</span></span></span></a><span key="aab1affa7e2b45418905b19ff5e41bc2"><span offset-key="aab1affa7e2b45418905b19ff5e41bc2:0">, </span></span><a href="https://en.wikipedia.org/wiki/Peer-to-peer" class="css-4rbku5 css-1dbjc4n r-1loqt21 r-1471scf r-1otgn73 r-1i6wzkk r-lrvibr"><span data-key="1b949c5129354512b819596629f1ad75" data-rnw-int-class="nearest_257-6651_259-6652-237__"><span data-key="f1253c188a6f4451a379298c85b61227"><span data-offset-key="f1253c188a6f4451a379298c85b61227:0">peer-to-peer networks</span></span></span></a><span key="a39b5c64ee9047c69dbb126a321ce116"><span offset-key="a39b5c64ee9047c69dbb126a321ce116:0">, etc.) are just there to provide a certain set of features and promises for the platform. Once you accept these features as given, you do not have to worry about the underlying technology - or do you have to know how Amazon’s AWS works internally in order to use it?</span></span>

### 

<span key="8a7933200bdd429689f8861f054617eb"><span offset-key="8a7933200bdd429689f8861f054617eb:0">Transactions</span></span>

<span key="fff38ca716374319b9bb2a2eeba47f9e"><span offset-key="fff38ca716374319b9bb2a2eeba47f9e:0">A blockchain is a globally shared, transactional database. This means that everyone can read entries in the database just by participating in the network. If you want to change something in the database, you have to create a so-called transaction which has to be accepted by all others. The word transaction implies that the change you want to make (assume you want to change two values at the same time) is either not done at all or completely applied. Furthermore, while your transaction is being applied to the database, no other transaction can alter it.</span></span>

<span key="be3f7ec54f514caf97c522b98991d70b"><span offset-key="be3f7ec54f514caf97c522b98991d70b:0">As an example, imagine a table that lists the balances of all accounts in an electronic currency. If a transfer from one account to another is requested, the transactional nature of the database ensures that if the amount is subtracted from one account, it is always added to the other account. If due to whatever reason, adding the amount to the target account is not possible, the source account is also not modified.</span></span>

<span key="865bf031349d4a609d8943315fd9f86e"><span offset-key="865bf031349d4a609d8943315fd9f86e:0">Furthermore, a transaction is always cryptographically signed by the sender (creator). This makes it straightforward to guard access to specific modifications of the database. In the example of the electronic currency, a simple check ensures that only the person holding the keys to the account can transfer money from it.</span></span>

### 

<span key="6e5328b707204497a8cfe665dc8a45a5"><span offset-key="6e5328b707204497a8cfe665dc8a45a5:0">Blocks</span></span>

<span key="600e57c2f7914f8ba17364803563aabb"><span offset-key="600e57c2f7914f8ba17364803563aabb:0">One major obstacle to overcome is what (in Bitcoin terms) is called a “double-spend attack”: What happens if two transactions exist in the network that both want to empty an account? Only one of the transactions can be valid, typically the one that is accepted first. The problem is that “first” is not an objective term in a peer-to-peer network.</span></span>

<span key="8e5362bcc7cc43de91278bfb564d3bcc"><span offset-key="8e5362bcc7cc43de91278bfb564d3bcc:0">The abstract answer to this is that you do not have to care. A globally accepted order of the transactions will be selected for you, solving the conflict. The transactions will be bundled into what is called a “block” and then they will be executed and distributed among all participating nodes. If two transactions contradict each other, the one that ends up being second will be rejected and not become part of the block.</span></span>

<span key="92799332fd094adea165afd035670e4e"><span offset-key="92799332fd094adea165afd035670e4e:0">These blocks form a linear sequence in time and that is where the word “blockchain” derives from. Blocks are added to the chain in rather regular intervals - for Ethereum this is roughly every 17 seconds.</span></span>

<span key="3fdad6bc869b49ad9acb0c9f7e107c96"><span offset-key="3fdad6bc869b49ad9acb0c9f7e107c96:0">As part of the “order selection mechanism” (which is called “mining”) it may happen that blocks are reverted from time to time, but only at the “tip” of the chain. The more blocks are added on top of a particular block, the less likely this block will be reverted. So it might be that your transactions are reverted and even removed from the blockchain, but the longer you wait, the less likely it will be.</span></span>

<span key="a903c62fc0444c70a982a31c2a48922c"><span offset-key="a903c62fc0444c70a982a31c2a48922c:0">Note</span></span>

<span key="b05c751c30104123b0e5c6c4faad73d1"><span offset-key="b05c751c30104123b0e5c6c4faad73d1:0">Transactions are not guaranteed to be included in the next block or any specific future block, since it is not up to the submitter of a transaction, but up to the miners to determine in which block the transaction is included.</span></span>

<span key="b0ff50bc97b640d3b8ee80b20d62b6f9"><span offset-key="b0ff50bc97b640d3b8ee80b20d62b6f9:0">If you want to schedule future calls of your contract, you can use the </span></span><a href="https://www.ethereum-alarm-clock.com" class="css-4rbku5 css-1dbjc4n r-1loqt21 r-1471scf r-1otgn73 r-1i6wzkk r-lrvibr"><span data-key="38920dc1d24148cc9dbfa6e0df235cdc" data-rnw-int-class="nearest_257-6651_259-6652-237__"><span data-key="8f1e0a369ee249d99e816ed233f833c1"><span data-offset-key="8f1e0a369ee249d99e816ed233f833c1:0">alarm clock</span></span></span></a><span key="2b4f5753ff6f4d3c8e3e7c9598647f5f"><span offset-key="2b4f5753ff6f4d3c8e3e7c9598647f5f:0"> or a similar oracle service.</span></span>

## 

<span key="618afb5fabfd4fcdb5b492295605be29"><span offset-key="618afb5fabfd4fcdb5b492295605be29:0">The Ethereum Virtual Machine</span></span>

### 

<span key="50b02ea99aa94c22a9100b34641be0a2"><span offset-key="50b02ea99aa94c22a9100b34641be0a2:0">Overview</span></span>

<span key="f30452ba0fa9480d9d9d73d5aa3a192a"><span offset-key="f30452ba0fa9480d9d9d73d5aa3a192a:0">The Ethereum Virtual Machine or EVM is the runtime environment for smart contracts in Ethereum. It is not only sandboxed but actually completely isolated, which means that code running inside the EVM has no access to network, filesystem or other processes. Smart contracts even have limited access to other smart contracts.</span></span>

### 

<span key="c17e4997abea4d5c85d7190ee261f68c"><span offset-key="c17e4997abea4d5c85d7190ee261f68c:0">Accounts</span></span>

<span key="59417b2ee88b453fb7fc3e5f1494f0af"><span offset-key="59417b2ee88b453fb7fc3e5f1494f0af:0">There are two kinds of accounts in Ethereum which share the same address space: </span>**External accounts**<span offset-key="59417b2ee88b453fb7fc3e5f1494f0af:2"> that are controlled by public-private key pairs (i.e. humans) and </span>**contract accounts**<span offset-key="59417b2ee88b453fb7fc3e5f1494f0af:4"> which are controlled by the code stored together with the account.</span></span>

<span key="f7876a6f75344473b6b02149e808c7e8"><span offset-key="f7876a6f75344473b6b02149e808c7e8:0">The address of an external account is determined from the public key while the address of a contract is determined at the time the contract is created (it is derived from the creator address and the number of transactions sent from that address, the so-called “nonce”).</span></span>

<span key="88d85a7487d449e99b1aee554f8b227b"><span offset-key="88d85a7487d449e99b1aee554f8b227b:0">Regardless of whether or not the account stores code, the two types are treated equally by the EVM.</span></span>

<span key="879075a9debd455bb3809fedc7444dac"><span offset-key="879075a9debd455bb3809fedc7444dac:0">Every account has a persistent key-value store mapping 256-bit words to 256-bit words called </span>**storage**<span offset-key="879075a9debd455bb3809fedc7444dac:2">.</span></span>

<span key="4440991ec5534e318d497399c0182b3b"><span offset-key="4440991ec5534e318d497399c0182b3b:0">Furthermore, every account has a </span>**balance**<span offset-key="4440991ec5534e318d497399c0182b3b:2"> in Ether (in “Wei” to be exact, </span>`1 ether`<span offset-key="4440991ec5534e318d497399c0182b3b:4"> is </span>`10**18 wei`<span offset-key="4440991ec5534e318d497399c0182b3b:6">) which can be modified by sending transactions that include Ether.</span></span>

### 

<span key="1d31be733d6c49c68b8c9832e935972b"><span offset-key="1d31be733d6c49c68b8c9832e935972b:0">Transactions</span></span>

<span key="9bc301c2f24441689b5f5ddd3fabdf8b"><span offset-key="9bc301c2f24441689b5f5ddd3fabdf8b:0">A transaction is a message that is sent from one account to another account (which might be the same or empty, see below). It can include binary data (which is called “payload”) and Ether.</span></span>

<span key="777789d8575d4e0fb13c0cc67935ea4c"><span offset-key="777789d8575d4e0fb13c0cc67935ea4c:0">If the target account contains code, that code is executed and the payload is provided as input data.</span></span>

<span key="7d989f255cf14a41b472ae5d7eda69df"><span offset-key="7d989f255cf14a41b472ae5d7eda69df:0">If the target account is not set (the transaction does not have a recipient or the recipient is set to </span>`null`<span offset-key="7d989f255cf14a41b472ae5d7eda69df:2">), the transaction creates a </span>**new contract**<span offset-key="7d989f255cf14a41b472ae5d7eda69df:4">. As already mentioned, the address of that contract is not the zero address but an address derived from the sender and its number of transactions sent (the “nonce”). The payload of such a contract creation transaction is taken to be EVM bytecode and executed. The output data of this execution is permanently stored as the code of the contract. This means that in order to create a contract, you do not send the actual code of the contract, but in fact code that returns that code when executed.</span></span>

<span key="64de43dfbbd64e199ae5a5e56858484a"><span offset-key="64de43dfbbd64e199ae5a5e56858484a:0">Note</span></span>

<span key="85aa3845cbe84e14840a6ccffc203ff0"><span offset-key="85aa3845cbe84e14840a6ccffc203ff0:0">While a contract is being created, its code is still empty. Because of that, you should not call back into the contract under construction until its constructor has finished executing.</span></span>

### 

<span key="84404511675e4da2889d16489e44f0e6"><span offset-key="84404511675e4da2889d16489e44f0e6:0">Gas</span></span>

<span key="f693adfda8fc48568585d05d57d8cf4b"><span offset-key="f693adfda8fc48568585d05d57d8cf4b:0">Upon creation, each transaction is charged with a certain amount of </span>**gas**<span offset-key="f693adfda8fc48568585d05d57d8cf4b:2">, whose purpose is to limit the amount of work that is needed to execute the transaction and to pay for this execution at the same time. While the EVM executes the transaction, the gas is gradually depleted according to specific rules.</span></span>

<span key="543bc457eeae4510ac899ceadbd12c7a"><span offset-key="543bc457eeae4510ac899ceadbd12c7a:0">The </span>**gas price**<span offset-key="543bc457eeae4510ac899ceadbd12c7a:2"> is a value set by the creator of the transaction, who has to pay </span>`gas_price * gas`<span offset-key="543bc457eeae4510ac899ceadbd12c7a:4"> up front from the sending account. If some gas is left after the execution, it is refunded to the creator in the same way.</span></span>

<span key="0983e38b54e149cf95e2bfc5dc21a6a9"><span offset-key="0983e38b54e149cf95e2bfc5dc21a6a9:0">If the gas is used up at any point (i.e. it would be negative), an out-of-gas exception is triggered, which reverts all modifications made to the state in the current call frame.</span></span>

### 

<span key="fa6f971778da4ec1bc44f1110e2dc48d"><span offset-key="fa6f971778da4ec1bc44f1110e2dc48d:0">Storage, Memory and the Stack</span></span>

<span key="a3e1b5f9ac964d2a8b9a90651c4c9658"><span offset-key="a3e1b5f9ac964d2a8b9a90651c4c9658:0">The Ethereum Virtual Machine has three areas where it can store data- storage, memory and the stack, which are explained in the following paragraphs.</span></span>

<span key="c8a4c428eb1846128be8901e8290ebb7"><span offset-key="c8a4c428eb1846128be8901e8290ebb7:0">Each account has a data area called </span>**storage**<span offset-key="c8a4c428eb1846128be8901e8290ebb7:2">, which is persistent between function calls and transactions. Storage is a key-value store that maps 256-bit words to 256-bit words. It is not possible to enumerate storage from within a contract, it is comparatively costly to read, and even more to initialise and modify storage. Because of this cost, you should minimize what you store in persistent storage to what the contract needs to run. Store data like derived calculations, caching, and aggregates outside of the contract. A contract can neither read nor write to any storage apart from its own.</span></span>

<span key="2d137651f62e4d06aa2180483fb3f5ef"><span offset-key="2d137651f62e4d06aa2180483fb3f5ef:0">The second data area is called </span>**memory**<span offset-key="2d137651f62e4d06aa2180483fb3f5ef:2">, of which a contract obtains a freshly cleared instance for each message call. Memory is linear and can be addressed at byte level, but reads are limited to a width of 256 bits, while writes can be either 8 bits or 256 bits wide. Memory is expanded by a word (256-bit), when accessing (either reading or writing) a previously untouched memory word (i.e. any offset within a word). At the time of expansion, the cost in gas must be paid. Memory is more costly the larger it grows (it scales quadratically).</span></span>

<span key="84894d5e8d5c4188ac5a628129998b50"><span offset-key="84894d5e8d5c4188ac5a628129998b50:0">The EVM is not a register machine but a stack machine, so all computations are performed on a data area called the </span>**stack**<span offset-key="84894d5e8d5c4188ac5a628129998b50:2">. It has a maximum size of 1024 elements and contains words of 256 bits. Access to the stack is limited to the top end in the following way: It is possible to copy one of the topmost 16 elements to the top of the stack or swap the topmost element with one of the 16 elements below it. All other operations take the topmost two (or one, or more, depending on the operation) elements from the stack and push the result onto the stack. Of course it is possible to move stack elements to storage or memory in order to get deeper access to the stack, but it is not possible to just access arbitrary elements deeper in the stack without first removing the top of the stack.</span></span>

### 

<span key="2ac32f46d84c4279890c58a42caa1507"><span offset-key="2ac32f46d84c4279890c58a42caa1507:0">Instruction Set</span></span>

<span key="5cfe72fe66dd4c10adb494a685ec3752"><span offset-key="5cfe72fe66dd4c10adb494a685ec3752:0">The instruction set of the EVM is kept minimal in order to avoid incorrect or inconsistent implementations which could cause consensus problems. All instructions operate on the basic data type, 256-bit words or on slices of memory (or other byte arrays). The usual arithmetic, bit, logical and comparison operations are present. Conditional and unconditional jumps are possible. Furthermore, contracts can access relevant properties of the current block like its number and timestamp.</span></span>

<span key="7b900459550a4102969efd3cc698aa28"><span offset-key="7b900459550a4102969efd3cc698aa28:0">For a complete list, please see the </span></span><a href="https://docs.soliditylang.org/en/v0.8.7/yul.html#opcodes" class="css-4rbku5 css-1dbjc4n r-1loqt21 r-1471scf r-1otgn73 r-1i6wzkk r-lrvibr"><span data-key="2e95572c8f744548babbe9cab8eb6805" data-rnw-int-class="nearest_257-6651_259-6652-237__"><span data-key="8397096d2c834d3385f2b0c7684fe70b"><span data-offset-key="8397096d2c834d3385f2b0c7684fe70b:0">list of opcodes</span></span></span></a><span key="08bcff5acca24bc5bb4d22f297329ad4"><span offset-key="08bcff5acca24bc5bb4d22f297329ad4:0"> as part of the inline assembly documentation.</span></span>

### 

<span key="8573fd9e7c46414f9e2c0dfd7cb88436"><span offset-key="8573fd9e7c46414f9e2c0dfd7cb88436:0">Message Calls</span></span>

<span key="581924655d3a4166b9f184ccbff5a689"><span offset-key="581924655d3a4166b9f184ccbff5a689:0">Contracts can call other contracts or send Ether to non-contract accounts by the means of message calls. Message calls are similar to transactions, in that they have a source, a target, data payload, Ether, gas and return data. In fact, every transaction consists of a top-level message call which in turn can create further message calls.</span></span>

<span key="28ad4228bfba499db3f9b49411208343"><span offset-key="28ad4228bfba499db3f9b49411208343:0">A contract can decide how much of its remaining </span>**gas**<span offset-key="28ad4228bfba499db3f9b49411208343:2"> should be sent with the inner message call and how much it wants to retain. If an out-of-gas exception happens in the inner call (or any other exception), this will be signaled by an error value put onto the stack. In this case, only the gas sent together with the call is used up. In Solidity, the calling contract causes a manual exception by default in such situations, so that exceptions “bubble up” the call stack.</span></span>

<span key="9da85844380f4394b53871b093a43d92"><span offset-key="9da85844380f4394b53871b093a43d92:0">As already said, the called contract (which can be the same as the caller) will receive a freshly cleared instance of memory and has access to the call payload - which will be provided in a separate area called the </span>**calldata**<span offset-key="9da85844380f4394b53871b093a43d92:2">. After it has finished execution, it can return data which will be stored at a location in the caller’s memory preallocated by the caller. All such calls are fully synchronous.</span></span>

<span key="3f5e10932d034874a882db21aa703b91"><span offset-key="3f5e10932d034874a882db21aa703b91:0">Calls are </span>**limited**<span offset-key="3f5e10932d034874a882db21aa703b91:2"> to a depth of 1024, which means that for more complex operations, loops should be preferred over recursive calls. Furthermore, only 63/64th of the gas can be forwarded in a message call, which causes a depth limit of a little less than 1000 in practice.</span></span>

### 

<span key="ddb5feb54207423f9a1d73fdcbf57c19"><span offset-key="ddb5feb54207423f9a1d73fdcbf57c19:0">Delegatecall / Callcode and Libraries</span></span>

<span key="b218130665354bd08feaf7093a656e6d"><span offset-key="b218130665354bd08feaf7093a656e6d:0">There exists a special variant of a message call, named </span>**delegatecall**<span offset-key="b218130665354bd08feaf7093a656e6d:2"> which is identical to a message call apart from the fact that the code at the target address is executed in the context of the calling contract and </span>`msg.sender`<span offset-key="b218130665354bd08feaf7093a656e6d:4"> and </span>`msg.value`<span offset-key="b218130665354bd08feaf7093a656e6d:6"> do not change their values.</span></span>

<span key="e2123fcde1084c3bbc02edf8c249f9ac"><span offset-key="e2123fcde1084c3bbc02edf8c249f9ac:0">This means that a contract can dynamically load code from a different address at runtime. Storage, current address and balance still refer to the calling contract, only the code is taken from the called address.</span></span>

<span key="4c9f2e7352734630ac16e76b46d1e285"><span offset-key="4c9f2e7352734630ac16e76b46d1e285:0">This makes it possible to implement the “library” feature in Solidity: Reusable library code that can be applied to a contract’s storage, e.g. in order to implement a complex data structure.</span></span>

### 

<span key="755234cf41af4bbbafcbbfe03a57ae02"><span offset-key="755234cf41af4bbbafcbbfe03a57ae02:0">Logs</span></span>

<span key="d9721860d99c4c379f57e5d5bae2ea04"><span offset-key="d9721860d99c4c379f57e5d5bae2ea04:0">It is possible to store data in a specially indexed data structure that maps all the way up to the block level. This feature called </span>**logs**<span offset-key="d9721860d99c4c379f57e5d5bae2ea04:2"> is used by Solidity in order to implement </span></span><a href="https://docs.soliditylang.org/en/v0.8.7/contracts.html#events" class="css-4rbku5 css-1dbjc4n r-1loqt21 r-1471scf r-1otgn73 r-1i6wzkk r-lrvibr"><span data-key="c498f1e1c5fb449da56b9d25874a9298" data-rnw-int-class="nearest_257-6651_259-6652-237__"><span data-key="750ec6091b7b48c7a0aaa09ba7ffaadc"><span data-offset-key="750ec6091b7b48c7a0aaa09ba7ffaadc:0">events</span></span></span></a><span key="39f9d70999c94237bbd39b8828d9398f"><span offset-key="39f9d70999c94237bbd39b8828d9398f:0">. Contracts cannot access log data after it has been created, but they can be efficiently accessed from outside the blockchain. Since some part of the log data is stored in </span></span><a href="https://en.wikipedia.org/wiki/Bloom_filter" class="css-4rbku5 css-1dbjc4n r-1loqt21 r-1471scf r-1otgn73 r-1i6wzkk r-lrvibr"><span data-key="831ea5e11a274db4a56527a9bab8688f" data-rnw-int-class="nearest_257-6651_259-6652-237__"><span data-key="9a9c42df8ebe4915a66fa8cf0734ae71"><span data-offset-key="9a9c42df8ebe4915a66fa8cf0734ae71:0">bloom filters</span></span></span></a><span key="6323b994a0794815a03300f8702a3a54"><span offset-key="6323b994a0794815a03300f8702a3a54:0">, it is possible to search for this data in an efficient and cryptographically secure way, so network peers that do not download the whole blockchain (so-called “light clients”) can still find these logs.</span></span>

### 

<span key="471f281440a04d05b2cf5b73235199e2"><span offset-key="471f281440a04d05b2cf5b73235199e2:0">Create</span></span>

<span key="62ca2882f87b48ad9d0491be8aae4994"><span offset-key="62ca2882f87b48ad9d0491be8aae4994:0">Contracts can even create other contracts using a special opcode (i.e. they do not simply call the zero address as a transaction would). The only difference between these </span>**create calls**<span offset-key="62ca2882f87b48ad9d0491be8aae4994:2"> and normal message calls is that the payload data is executed and the result stored as code and the caller / creator receives the address of the new contract on the stack.</span></span>

### 

<span key="2f8afd2aaff549babf793698988f80c3"><span offset-key="2f8afd2aaff549babf793698988f80c3:0">Deactivate and Self-destruct</span></span>

<span key="c2fe686e8637416dabef7d3494a25aab"><span offset-key="c2fe686e8637416dabef7d3494a25aab:0">The only way to remove code from the blockchain is when a contract at that address performs the </span>`selfdestruct`<span offset-key="c2fe686e8637416dabef7d3494a25aab:2"> operation. The remaining Ether stored at that address is sent to a designated target and then the storage and code is removed from the state. Removing the contract in theory sounds like a good idea, but it is potentially dangerous, as if someone sends Ether to removed contracts, the Ether is forever lost.</span></span>

<span key="6b2e301f59d848ed86c37eb660f60c72"><span offset-key="6b2e301f59d848ed86c37eb660f60c72:0">Warning</span></span>

<span key="43206456d6384c55a23813c4fc955272"><span offset-key="43206456d6384c55a23813c4fc955272:0">Even if a contract is removed by </span>`selfdestruct`<span offset-key="43206456d6384c55a23813c4fc955272:2">, it is still part of the history of the blockchain and probably retained by most Ethereum nodes. So using </span>`selfdestruct`<span offset-key="43206456d6384c55a23813c4fc955272:4"> is not the same as deleting data from a hard disk.</span></span>

<span key="b7229f4730634aeabaca3652c142f1dd"><span offset-key="b7229f4730634aeabaca3652c142f1dd:0">Note</span></span>

<span key="290d50881c0d4bae922b8eb9fc2513cc"><span offset-key="290d50881c0d4bae922b8eb9fc2513cc:0">Even if a contract’s code does not contain a call to </span>`selfdestruct`<span offset-key="290d50881c0d4bae922b8eb9fc2513cc:2">, it can still perform that operation using </span>`delegatecall`<span offset-key="290d50881c0d4bae922b8eb9fc2513cc:4"> or </span>`callcode`<span offset-key="290d50881c0d4bae922b8eb9fc2513cc:6">.</span></span>

<span key="a1bfdd9b116b4884bbfebd0a328065de"><span offset-key="a1bfdd9b116b4884bbfebd0a328065de:0">If you want to deactivate your contracts, you should instead </span>**disable**<span offset-key="a1bfdd9b116b4884bbfebd0a328065de:2"> them by changing some internal state which causes all functions to revert. This makes it impossible to use the contract, as it returns Ether immediately.</span></span><a href="https://docs.soliditylang.org/en/v0.8.7/installing-solidity.html" class="css-4rbku5 css-1dbjc4n r-1loqt21 r-1471scf r-1otgn73 r-1i6wzkk r-lrvibr"><span data-key="f4c4479c9cd24f3bb815d002000adcd9" data-rnw-int-class="nearest_257-6651_259-6652-237__"><span data-key="18cd69a516434ce59f99739c90b24106"><span data-offset-key="18cd69a516434ce59f99739c90b24106:0">Next </span></span></span></a><span key="2c78d3233f4b4b908fadd8e118d2ae51"><span offset-key="2c78d3233f4b4b908fadd8e118d2ae51:0"><span slate-zero-width="z">​</span></span></span><a href="https://docs.soliditylang.org/en/v0.8.7/index.html" class="css-4rbku5 css-1dbjc4n r-1loqt21 r-1471scf r-1otgn73 r-1i6wzkk r-lrvibr"><span data-key="6fb8a63683954a43b051071632028b0e" data-rnw-int-class="nearest_257-6651_259-6652-237__"><span data-key="6763e7e101df4ad0a26d4e1bdea3a3cb"><span data-offset-key="6763e7e101df4ad0a26d4e1bdea3a3cb:0"> Previous</span></span></span></a><span key="81223e8d75af492095941d0714463f2c"><span offset-key="81223e8d75af492095941d0714463f2c:0"> </span></span>
